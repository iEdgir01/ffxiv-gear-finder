# Technical Context

## Stack
- Plain HTML/CSS/ES Modules — no bundler, no build step
- Open index.html directly in browser (or `python -m http.server 3000`)
- Tests: `npm test` → `node --test tests/*.test.js` — 147 tests across `tests/` covering search, api, lists, garland, upgrade, gearBaseline, gearsets, constants, gcSealCost, finderSourceFilter
- **Deployed**: GitHub Pages at https://iedgir01.github.io/ffxiv-gear-finder/ (auto-deploys on push to main via `.github/workflows/deploy.yml`)
- **Datamine CI**: `.github/workflows/weekly-datamine-refresh.yml` — Mondays 06:00 UTC, fetches xivapi/ffxiv-datamining CSVs, regenerates `js/gcData.js` + `js/specialVendorData.js`, runs tests, commits + pushes if changed (which triggers redeploy)

## Data Sources

### Teamcraft (recipe data, loaded at startup)
Base URL: `https://raw.githubusercontent.com/ffxiv-teamcraft/ffxiv-teamcraft/master/libs/data/src/lib/json/`
- `recipes.json` — array of recipe objects: `{ id, job, lvl, rlvl, result, yields, ingredients }`
- `items.json` — object keyed by item ID string: `{ "6116": { "en": "Rose Gold Ring", ... } }`

Field names confirmed: `job` (integer), `lvl` (craft level), `result` (item ID integer).

Loaded once at startup, indexed in memory. ~2-5MB total.

### XIVAPI (character import + item stats)
- Character search: `GET https://xivapi.com/character/search?name={name}&server={server}`
- Character jobs: `GET https://xivapi.com/character/{id}?data=CJ`
- Item stats: `GET https://xivapi.com/search?indexes=Item&filters=ID|={id1,id2,...}&columns=ID,Name,LevelItem,LevelEquip,Stats,ItemUICategory,ClassJobCategory`

Stats come back as an object with string keys matching our STATS_BY_GROUP keys (PascalCase: "CriticalHit", "DirectHitRate", "SkillSpeed", "SpellSpeed", "Strength", etc.).

### Grand Company data (`js/gcData.js` — generated)

Generated by `scripts/build-gc-data.mjs` from **xivapi/ffxiv-datamining** CSVs: `SpecialShop.csv`, `Item.csv`, `GrandCompanyRank.csv`, `ClassJobCategory.csv`, `ItemUICategory.csv`, `BaseParam.csv`. Stat names are normalized in the script (e.g. “Direct Hit Rate” → `DirectHitRate`) to match XIVAPI / `STATS_BY_GROUP`.

Classic seal rows use `SpecialShop` with `CostType === 0` and seal currency ids **20 / 21 / 22**. Some artisan tools are listed under Talan’s **Artisan Tools (DoH)** row with a different cost shape; for those item ids the build **fetches Garland Tools** JSON at build time and reads seal cost from **Grand Company quartermaster** listings only (see below).

- Output: `GC_ITEMS` keyed by item id — `seals`, `companyId`, `currencyId`, equip/ilvl, stats, quest/rank metadata where present.
- Regenerate: `npm run build:gc-data` — **do not hand-edit seal amounts** in `gcData.js`.

#### External data policy (no hand-maintained prices)

Numeric costs and item attributes come from **external** sources only. We do not maintain price override tables in the repo.

| Data | Source |
|------|--------|
| Classic GC rows in `SpecialShop` | Datamining CSVs (via `build-gc-data.mjs`) |
| Seal cost for artisan tools where CSV ≠ quartermaster seals | **Garland** item JSON at build time (same family of API as runtime `garland.js`) |
| Card stats | **XIVAPI** (`api.js`) |
| Acquisition tags | **Garland** at runtime (`garland.js`) |

Our code does **parsing, joins, and normalization** — not overwriting source numbers.

#### Why Garland GC supplement uses **quartermaster lines only**

Garland `tradeShops` for one item can include multiple shops (e.g. **Artisan Tools (DoH)** / Talan with **token** currencies, plus **Maelstrom**, **Order of the Twin Adder**, **Immortal Flames** quartermasters with **company seals**). This app only needs **“how many company seals at a GC quartermaster?”** So the build filters Garland to the three quartermaster shop names and ignores other shops on the same item. **We only take quartermaster lines so we only record seal costs from quartermasters.**

Integration:
- `data.js` merges `GC_ITEMS` into the pool with `gcInfo`, etc.
- `main.js` uses GC metadata for Source → GC and Include GC.
- `ui.js` displays seals + rank where present.

### Garland Tools (pure classification only — no runtime fetch)
- `js/garland.js` retains pure functions: `parseGarlandDoc`, `classifyAcquisition`, `isGcExclusiveAcquisition`, `syntheticAcqFromItem`.
- **No runtime network calls** — fetch infrastructure removed. Source classification is synchronous from local item metadata (`gcInfo`, `tomestoneInfo`, `scripInfo`).
- `state.acqCache` still exists but is populated from local data only (no Garland API calls).
- Source tags and Source filter work without any external requests.

## Search / Scoring

### Gear Finder filter chain
1. `getItemsInLevelRange` — pool from Teamcraft recipes + GC items in recipe level band.
2. `filterItems` — recipe level, equip level, gear type, job equip constraint (`ClassJobCategory`).
3. `filterOutGcExclusiveItems` — if Include GC is off, removes GC-only items.
4. `filterItemsByAcquisition` — applies Source filter (all / gc / craft).
5. `sortGearForDisplay` — sorts by priority stat or sum of group stats (see Scoring).
6. Live search filter — substring match on item name (lowercase), applied in `runSearch` before render.

### Scoring (sum-based)
`maxGroupStatScore(item, groupKey)` returns the **sum** of all group-relevant stats present on the item (positive values only). Returns `-Infinity` if no relevant stats found.

All groups use sum scoring: DoH (CP + Craftsmanship + Control), DoL (GP + Gathering + Perception), DoW/DoM (primary + secondary combat stats).

### Priority stat dropdown
`state.priorityStat` — null = "Best overall" (sum), string = sort by that single stat descending.
Options populated via `collectPriorityStatOptions` from stats actually present on the filtered result set.
Display labels use `statLabel()` / `STAT_DISPLAY_NAMES` in `ui.js` (e.g. "CriticalHit" → "Critical Hit").

### Top pick (red dot)
`findTopPickIdsByMaxGroupStat` — best per slot when all gear types shown; best in list when one gear type selected. Uses priority stat if set, else sum score.

### Job equip filter
`jobCanEquipCategory(jobId, categoryName)` maps XIVAPI `ClassJobCategory` strings → job abbrs via `CLASSJOB_CATEGORY_TO_JOBS`. Recipe crafter (`craftJobAbbr`) is display-only.

**Important:** `specialVendorData.js` (tomestone/scrip items) uses space-separated abbreviation lists for `classJobCategory` (e.g. `"GLA MRD PLD WAR DRK GNB"`), not the CLASSJOB_CATEGORY_TO_JOBS keyword format. `jobCanEquipCategory` handles both: single-keyword lookup first, then space-separated abbreviation list split.

## Character / Profile System

### Storage
`js/profiles.js` — localStorage key `gf_profiles_v1`. Stores `{ version, activeLodestoneId, profiles: Record<id, StoredProfile> }`. Each profile: `lodestoneId`, `name`, `server`, `portrait`, `jobs`, `teamcraftUid`, `teamcraftProfileUrl`, `masterStars`, include-toggles, `upgradeSourceMode`.

### Character overlay — two-screen design
- **Manage screen** (`#char-screen-manage`): profile cards listing all saved characters. Each card: portrait, name/server (read-only), editable Teamcraft URL + Save button, Use (disabled if active) and Remove buttons.
- **Add screen** (`#char-screen-add`): import form (name/DC/server search). Back button hidden when no profiles exist.
- Logic: open overlay → manage if profiles exist, add if none. Successful import → auto-switch to manage. Last profile removed → switch to add. `ui.resetAddForm()` called whenever add screen shown (clears DC/server/name/results).

### On-load refresh (`refreshCharacterJobsOnLoad`)
1. Fetch Lodestone job levels (with one retry).
2. If `state.uid` is set, also fetch Teamcraft job stats via `fetchByTeamcraftUID`.
3. Merge: take `max(Lodestone level, Teamcraft level)` per job — Teamcraft reflects in-game progress faster than Lodestone.
4. Persist merged levels to profile, re-run search + upgrades.
5. Re-fetch Teamcraft gearsets if `state.uid` is set.

### Per-card Teamcraft linking
`handleTcSaveForCard(lodestoneId, url)` saves TC URL to the profile. If the card is the currently active profile, also loads gearsets and re-runs upgrades + search.

## Lists & Teamcraft Export
Lists are saved in localStorage via `js/lists.js`.
Teamcraft export format: `id,null,qty;id,null,qty` (semicolons, no trailing), base64 via `btoa()`.
URL: `https://ffxivteamcraft.com/import/{base64string}`

## Live Search
`#finder-search` input — filters by item name substring (case-insensitive), applied client-side after sorting.
`state.searchQuery` holds the normalized query. `clearSearch()` in `main.js` resets state + DOM input; called on job group change and job change.

## Module Boundaries
- `constants.js` — data only, no functions
- `search.js` — pure functions, no DOM, no fetch
- `data.js` — fetch + in-memory cache, no DOM
- `api.js` — fetch only, no DOM, no caching
- `garland.js` — fetch + classification, no DOM
- `upgrade.js` — pure upgrade logic, no DOM, no fetch
- `gearBaseline.js` — baseline stat resolution, no DOM, no fetch
- `gearsets.js` — Teamcraft gearset fetch, no DOM
- `lists.js` — localStorage CRUD + Teamcraft URL builder, no DOM
- `gcData.js` — generated constant, data only
- `ui.js` — DOM only, emits events, does not call search.js or api.js directly
- `main.js` — orchestrates all others, owns all state
